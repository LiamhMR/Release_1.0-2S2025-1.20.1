Plan de Pruebas para la Aplicación de Pre-Test y Post-Test

11.1. Propósito y objetivo del plan de pruebas

El presente plan de pruebas tiene como propósito definir la forma en que se evaluará la experiencia de uso del entorno educativo implementado en Minecraft, particularmente en lo relativo a la aplicación de un pre-test y un post-test asociados al uso de las modalidades SQL del servidor. Este documento no se limita a describir la ejecución de cuestionarios, sino que también establece las condiciones pedagógicas y técnicas bajo las cuales dichas pruebas serán aplicadas.

La necesidad de formular este plan surge de una consideración metodológica importante: el grupo de prueba potencial puede incluir estudiantes de Ingeniería Civil Industrial, quienes no necesariamente poseen el mismo alcance formativo en SQL que un grupo especializado en informática. Por ello, el entorno experimental debe ajustarse para evaluar comprensión y adopción de mecánicas educativas sin exigir niveles de complejidad que excedan el alcance razonable del público participante.

Adicionalmente, el objetivo de esta fase no es volver a demostrar si es factible aprender mediante un juego diseñado dentro de Minecraft, puesto que dicho aspecto ya fue validado en la etapa anterior del seminario. Lo que se busca ahora es validar si la modalidad específica que ha sido creada cumple efectivamente ese propósito, es decir, si el diseño concreto implementado en SQL Dungeon, SQL Battle y el sistema de pre-test y post-test resulta adecuado como medio de apoyo al aprendizaje y evaluación de contenidos SQL.

11.2. Criterio pedagógico de delimitación

Para que la prueba sea válida y no se vea afectada por una dificultad técnica excesiva ajena al objetivo del estudio, se definió una reducción controlada del contenido SQL que será exigido a los participantes. Esta delimitación responde directamente al perfil del grupo de prueba y busca medir la interacción con la plataforma sin convertir la experiencia en una barrera por sobrecarga conceptual.

ModalidadDelimitación definida para la pruebaJustificación
SQL Dungeon Se limitará la experiencia a consultas correspondientes hasta el nivel 3 del banco de preguntas. Esto permite trabajar con contenidos introductorios e intermedios sin exigir operaciones más avanzadas como INNER JOIN, reservadas para etapas posteriores.
SQL Battle Si bien técnicamente pueden existir cruces más complejos, el modelo ER fue diseñado para que, en condiciones normales de uso, no sea necesario cruzar más de dos tablas en una misma consulta. Se mantiene el componente relacional básico del juego, pero el diseño de la base favorece consultas comprensibles para el grupo de prueba y evita exigir complejidad innecesaria.

11.3. Preparación del entorno de SQL Dungeon

En la modalidad SQL Dungeon, el entorno de prueba será ajustado para utilizar exclusivamente los desafíos incluidos hasta el nivel 3 del banco de preguntas, contenido que se referenciará posteriormente en el anexo del informe. Esta decisión significa que la prueba no utilizará los niveles más avanzados, donde aparecen estructuras relacionales de mayor dificultad o problemas que requieren competencias SQL más desarrolladas.

De esta forma, el recorrido evaluado se concentrará en operaciones de consulta, filtrado e interpretación de resultados dentro de un nivel de dificultad coherente con un usuario que se encuentra en una etapa inicial o intermedia de aprendizaje. El objetivo no es medir dominio experto de SQL, sino verificar si el entorno lúdico favorece la comprensión, motivación y continuidad en la interacción con el contenido.

11.3.1. Alcance esperado en SQL Dungeon

  • Consultas SELECT sobre una sola tabla.
  • Uso de filtros simples mediante WHERE.
  • Interpretación de resultados devueltos por el sistema.
  • Resolución de enunciados hasta el nivel 3 del banco de preguntas.
  • Exclusión explícita de desafíos avanzados que requieran INNER JOIN.

11.4. Preparación del entorno de SQL Battle

En SQL Battle, la experiencia no será simplificada al punto de eliminar el componente relacional, ya que una parte importante de esta modalidad consiste precisamente en consultar el estado de los recursos disponibles y tomar decisiones tácticas en función de esa información. Sin embargo, el entorno fue estructurado para que, aun cuando sean posibles consultas más complejas, las acciones fundamentales de la modalidad puedan resolverse normalmente con relaciones sencillas entre tablas, especialmente mediante cruces de hasta dos tablas.

Este criterio permite que el jugador experimente la lógica relacional sin verse obligado a formular consultas de alta complejidad. Además, el diseño de SQL Battle ofrece un mecanismo complementario relevante: como las respuestas del sistema se entregan en forma de libros, el jugador puede consultar tablas por separado y cruzar la información manualmente si así lo necesita. Esto reduce la carga cognitiva sin invalidar la experiencia.

11.4.1. Consulta base de referencia para recursos

Una consulta representativa de la modalidad, útil para revisar la cantidad de recursos disponibles, es la siguiente:

SELECT id,nombre,cantidad FROM tipos_item JOIN Almacen ON item_id=id

Esta consulta resume una operación relacional básica entre dos tablas, suficiente para ilustrar el tipo de uso esperado durante la prueba. No obstante, también será posible resolver la misma necesidad mediante consultas separadas, por ejemplo consultando primero tipos_item y luego Almacen, para que el jugador cruce los resultados manualmente a partir de los libros de respuesta entregados por el sistema.

11.4.2. Estrategia alternativa con menor complejidad

En caso de que el participante no desee o no logre utilizar un JOIN, podrá recurrir a una estrategia en dos pasos como la siguiente:

  • Ejecutar SELECT * FROM tipos_item para identificar el id del recurso buscado.
  • Ejecutar SELECT * FROM Almacen para revisar la cantidad asociada a dicho id.
  • Cruzar manualmente ambos resultados usando los libros generados por el juego.

Este comportamiento no se considerará incorrecto dentro del contexto de la prueba, ya que el propósito en esta etapa no es exigir optimización de consultas, sino verificar la capacidad del jugador para obtener y utilizar información dentro del entorno propuesto.

11.4.3. Consulta base de referencia para inserción

Durante la primera prewave, el sistema entrega sugerencias de consultas al jugador. Una de las más importantes para el plan de pruebas es la siguiente:

INSERT INTO inventario (item_id,cantidad,activo_en_etapa) VALUES (1,1,1)

Esta consulta cumple una función tutorial clave, ya que muestra de forma explícita cómo insertar elementos del almacén dentro del inventario del jugador. Los significados de los campos item_id, cantidad y activo_en_etapa se explican dentro del tutorial del juego, y además el participante puede recurrir al NPC Harry para obtener orientación adicional cuando lo requiera.

11.4.4. Progresión esperada dentro de SQL Battle

A partir de estas consultas base, el jugador podrá comenzar a desenvolverse en la modalidad. No obstante, el progreso dentro del juego seguirá requiriendo ajustes progresivos en sus consultas, por ejemplo para explorar también los enemigos de la oleada o decidir qué recursos reservar para etapas posteriores. En otras palabras, el plan de pruebas considera que el sistema debe permitir un inicio accesible, pero también dejar espacio para que la experiencia y la observación ayuden al jugador a refinar su estrategia conforme avanza.

11.5. Preparación de infraestructura y acceso

  • Para las semanas de prueba se preparará el host para soportar hasta 30 jugadores simultáneos, con un tope de 45, aumentando recursos de procesador y memoria RAM. Como apoyo a esta carga, se dispondrán dos arenas paralelas por modalidad, tanto para SQL Dungeon como para SQL Battle.
  • La prueba también considera la ventaja práctica del ecosistema abierto de Minecraft. Aunque Mojang mantiene sus términos de uso, en la práctica ha favorecido una comunidad amplia mediante configuraciones como online-mode=false, lo que ha permitido el acceso desde contextos donde el uso no premium es común. En este escenario se contempla el uso de launchers validados por la comunidad, como SKLauncher y Fenix, para facilitar la participación desde distintos dispositivos y también desde los hogares de los estudiantes.
  • Se pondrá a disposición de los participantes toda la documentación y los archivos necesarios para instalar el juego en la versión correspondiente y agregar el servidor. Este proceso es relativamente sencillo y permitirá reducir barreras de acceso antes de la aplicación formal de las pruebas.

11.6. Estructura general de la aplicación de pruebas

La aplicación se organizará en una secuencia de tres momentos: un levantamiento inicial mediante pre-test, una fase de interacción con las modalidades del servidor y una recogida posterior mediante post-test. En esta versión del documento se deja establecida la estructura general, mientras que los detalles operativos específicos de los instrumentos podrán agregarse en una iteración posterior.

  1. Aplicación del pre-test antes de comenzar la experiencia de juego.
  2. Ingreso del participante al entorno y desarrollo de la actividad con las restricciones aquí definidas.
  3. Aplicación del post-test al finalizar el recorrido experimental.

11.7. Consideraciones metodológicas para el pre-test y el post-test

Tanto el pre-test como el post-test deberán diseñarse en coherencia con el alcance realmente exigido durante la experiencia. En consecuencia, si el entorno de prueba no utilizará consultas avanzadas con múltiples relaciones, entonces los instrumentos tampoco deberán centrarse en evaluar ese nivel de complejidad. Esto resulta especialmente importante si la muestra incluye estudiantes ajenos a una formación estrictamente informática.

En este sentido, el plan propone que los instrumentos prioricen aspectos como comprensión básica de consultas, lectura de resultados, percepción de dificultad, utilidad percibida del entorno y disposición a seguir aprendiendo mediante experiencias gamificadas.

11.7.1. Criterio de coherencia entre instrumento y entorno

  • El pre-test debe medir conocimientos previos compatibles con el alcance de la prueba.
  • El post-test debe recoger tanto percepción de experiencia como cambios observables en comprensión básica.
  • No se evaluarán contenidos que no hayan sido realmente puestos en juego durante la actividad.

11.9. Resultado esperado del plan de pruebas

Se espera que, bajo estas condiciones de ajuste, los participantes puedan interactuar con el entorno educativo sin que la complejidad de SQL se convierta en un obstáculo desproporcionado. El interés principal de esta fase es verificar si el servidor logra funcionar como medio didáctico comprensible, accesible y motivador, antes de escalar la dificultad hacia contenidos más complejos.

En consecuencia, este plan de pruebas actúa como una etapa de validación controlada, donde se privilegia la pertinencia pedagógica del entorno y la experiencia del usuario por sobre la cobertura total del contenido técnico.

11.10. Secciones pendientes de completar

Este documento deja preparada la base del plan de pruebas, especialmente en lo relativo a la delimitación del entorno, la justificación pedagógica del recorte y los criterios de observación. En una siguiente iteración se podrán incorporar con mayor detalle los siguientes elementos:

  • Diseño específico del pre-test.
  • Diseño específico del post-test.
  • Procedimiento de convocatoria y aplicación con los participantes.
  • Cantidad de estudiantes, duración estimada y condiciones de ejecución.
  • Formato de registro y análisis de resultados.
